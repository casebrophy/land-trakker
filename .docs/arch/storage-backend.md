# Storage Backend System

PostgreSQL persistence layer with sqlc code generation. Three domain stores (`sourcedb.Store`, `listingdb.Store`, `searchdb.Store`) implement domain Storer interfaces (`source.Storer`, `listing.Storer`, `search.Storer`). Schema defines sources/scrape_runs/raw_fetches for web scraping; listings/snapshots/price_changes/possible_duplicates/auction_extension for tracked land properties; and saved_searches/search_hits for user search curation. All code in `storage/db/` is generated by sqlc from migration schemas and SQL queries.

### ⚠ QueryListingsFilter() Method (storage/listingdb/store.go:312–441)
**Dynamic SQL filtering with `listing.ListingFilter` struct — custom implementation, not sqlc-generated.**
- Accepts `ListingFilter` with optional fields: `AcresMin/Max`, `PriceMin/Max`, `Counties`, `PPAMin/Max`, `PropertyType`, boolean attributes
- Builds WHERE clause conditionally: `acres >= $N`, `price_cents >= $N`, `county = ANY($N)`, etc.
- Scans results into `db.GetListingByIDRow` and converts to `listing.Listing` using `getByIDRowToListing()`
- **Impact:** Changing `ListingFilter` struct requires updating the filter builder logic in lines 342–380
- **Integration test:** `storage/listingdb/listingdb_integration_test.go:TestQueryListingsFilter` validates all filter combinations

### ⚠ SavedSearches & SearchHits Tables (0004_saved_searches.sql)
**User-curated search filters stored as ListingFilter (jsonb) with dedup hit tracking.**
- `saved_searches`: id (uuid), name, query (ListingFilter as jsonb), created_at, enabled (boolean)
- `search_hits`: id (bigserial), saved_search_id (fk), listing_id (fk), hit_at, reason (text), seen (boolean)
- **Unique constraint:** (saved_search_id, listing_id, reason, hit_at) prevents duplicate hit records
- **Index:** search_hits (hit_at DESC) WHERE seen=false for unseen query optimization
- **Implementation:** `storage/searchdb/store.go` — raw pgx (not sqlc), marshalFilter/unmarshalFilter handle json ⇄ domain.ListingFilter conversion
- **Callers:** business/domain/search/* services for saved search CRUD and hit workflow

### ⚠ PossibleDuplicates Table (0005_possible_duplicates.sql)
**Pair-wise listing deduplication with user merge decisions.**
- Columns: listing_a_id (uuid, fk), listing_b_id (uuid, fk), score (numeric 0.00–1.00), reasons (text[]), detected_at (timestamptz), user_decision (text, nullable)
- **Primary key:** (listing_a_id, listing_b_id) with CHECK (listing_a_id < listing_b_id) enforces canonical ordering
- **Implementation:** `storage/listingdb/store.go` — QueryListingsForDedup (list candidates), UpsertPossibleDuplicate (upsert pair), QueryPossibleDuplicates (filter by decision status)
- **Callers:** business/domain/listing/* dedup service for similarity detection and merge workflows
- **No sqlc queries:** Custom implementation in listingdb for dynamic pair management

### ⚠ AuctionExtension Table (0006_auction_extension.sql)
**Auction metadata split from main listings table for optional per-property tracking.**
- Columns: id (bigserial), listing_id (uuid, UNIQUE fk), auction_end_date (timestamptz, nullable), auction_current_bid (bigint, nullable), auction_reserve (bigint, nullable)
- **Index:** listing_id for fast lookup by listing
- **Implementation:** `storage/listingdb/store.go` — CreateAuctionExt, UpdateAuctionExt, getAuctionExtByListingID, getAuctionExtByListingIDs; sqlc queries in `storage/queries/listing.sql` (CreateAuctionExt, GetAuctionExt, UpdateAuctionExt)
- **Integration:** listingdb.CreateListing/UpdateListing check if auction fields non-nil, then create/update auction_extension row; QueryListingByID and QueryListings join auction_extension via getAuctionExtByListingID(s)
- **Callers:** business/domain/listing/* for auction property tracking

### ⚠ Schema Changes (0001_core.sql, 0002_listings.sql, Phase 2 migrations)
**Any table or column changes require coordinated updates across the stack.**
1. Create new migration file in `storage/migrations/` (e.g., `0007_add_field.sql`)
2. If using sqlc: update `.sql` queries in `storage/queries/` that reference changed tables; run `make sqlc-generate`
3. If raw pgx (search/dedup): hand-write store methods in `storage/{domain}db/store.go`
4. Update domain model types in `business/domain/{feature}/*` if struct fields changed
5. Update store converters and type mapping logic for db ⇄ domain conversions
6. Verify all business layer code expecting old types is updated

### ⚠ Cross-Domain Type Conversions
**Store implementations map between sqlc-generated pgx types and domain types.**
- `storage/sourcedb/store.go` — rowToSource(), rowToRun(), rowToRawFetch() convert db.* ⇄ source.*
- `storage/listingdb/store.go` — createRowToListing(), getByIDRowToListing(), rowToSnapshot(), rowToPriceChange() convert db.* ⇄ listing.*
- **Change impact:** Modifying domain struct field names or types requires updating corresponding converter functions

## Cross-Domain Dependencies

**Outbound:** None. Storage is the leaf layer.

**Inbound:**
- `business/domain/source/*` — depends on source.Storer interface (implemented by sourcedb.Store)
- `business/domain/listing/*` — depends on listing.Storer interface (implemented by listingdb.Store); uses dedup and auction extension functionality
- `business/domain/search/*` — depends on search.Storer interface (implemented by searchdb.Store)
- `app/domain/sourceapp/*` — indirectly, via source business domain
- `app/domain/listingapp/*` — indirectly, via listing business domain
- `app/domain/searchapp/*` — indirectly, via search business domain

**Initialization:** All stores created via `NewStore(pool *pgxpool.Pool)` and registered as dependency injection in respective business domain constructors.
