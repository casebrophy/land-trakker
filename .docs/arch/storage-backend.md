# Storage Backend System

PostgreSQL persistence layer with sqlc code generation. Two domain stores (`sourcedb.Store`, `listingdb.Store`) implement domain Storer interfaces (`source.Storer`, `listing.Storer`). Schema defines sources/scrape_runs/raw_fetches for web scraping, and listings/snapshots/price_changes for tracked land properties. All code in `storage/db/` is generated by sqlc from migration schemas and SQL queries.

## File Map

## Impact Callouts

### ⚠ storage/db/* (Auto-Generated Code)
**These files are generated by sqlc and MUST NOT be hand-edited.**
- Regeneration: `make sqlc-generate` in repo root
- Triggered by changes to: `storage/migrations/*.sql` or `storage/queries/*.sql`
- Always regenerate before commit if any .sql files changed
- Changes propagate to all callers of exported sqlc types and methods

### ⚠ source.Storer Interface
**Changing interface methods directly impacts all implementations and callers.**
- **Implementation:** `storage/sourcedb/store.go` — must implement all methods
- **Callers:** `business/domain/source/*` services that depend on source.Storer
- **Ripple:** Adding/removing a method requires corresponding .sql query changes and sqlc regeneration

### ⚠ listing.Storer Interface
**Changing interface methods directly impacts all implementations and callers.**
- **Implementation:** `storage/listingdb/store.go` — must implement all methods
- **Callers:** `business/domain/listing/*` services that depend on listing.Storer
- **Ripple:** Adding/removing a method requires corresponding .sql query changes and sqlc regeneration

### ⚠ Schema Changes (0001_core.sql, 0002_listings.sql)
**Any table or column changes require coordinated updates across the stack.**
1. Create new migration file in `storage/migrations/` (e.g., `0003_add_field.sql`)
2. Update `.sql` queries in `storage/queries/` that reference changed tables
3. Run `make sqlc-generate` to regenerate `storage/db/*`
4. Update domain model types in `business/domain/source/*` and `business/domain/listing/*` if struct fields changed
5. Update store converters in `storage/sourcedb/store.go` or `storage/listingdb/store.go` (type mapping logic)
6. Verify all business layer code expecting old types is updated

### ⚠ Cross-Domain Type Conversions
**Store implementations map between sqlc-generated pgx types and domain types.**
- `storage/sourcedb/store.go` — rowToSource(), rowToRun(), rowToRawFetch() convert db.* ⇄ source.*
- `storage/listingdb/store.go` — createRowToListing(), getByIDRowToListing(), rowToSnapshot(), rowToPriceChange() convert db.* ⇄ listing.*
- **Change impact:** Modifying domain struct field names or types requires updating corresponding converter functions

## Cross-Domain Dependencies

**Outbound:** None. Storage is the leaf layer.

**Inbound:**
- `business/domain/source/*` — depends on source.Storer interface (implemented by sourcedb.Store)
- `business/domain/listing/*` — depends on listing.Storer interface (implemented by listingdb.Store)
- `app/domain/sourceapp/*` — indirectly, via source business domain
- `app/domain/listingapp/*` — indirectly, via listing business domain

**Initialization:** Both stores created via `NewStore(pool *pgxpool.Pool)` and registered as dependency injection in business domain constructors.
